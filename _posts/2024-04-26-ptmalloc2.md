## ptmalloc2
모든 프로세스는 실행 중에 메모리를 동적으로 할당하고, 할당한 메모리의 쓰임이 다하면 메모리를 해제한다. 

**Memory Allocator**는 사용된 알고리즘에 따라 여러 종류가 있는데, 리눅스는 **`ptmalloc`**과 **`dlmalloc`**, 크롬은 **`tcmalloc`**, 파이어폭스는 **`jemalloc`** 등이 있다.

**ptmalloc2**는 어떤 메모리가 해제되면, 해제된 메모리의 특징을 기억하고 있다가 비슷한 메모리의 할당 요청이 발생하면 이를 빠르게 반환한다. 이를 통해 **메모리 할당의 속도를 높이고** **불필요한 메모리 사용을 막을 수 있고**, **메모리 단편화를 방지할 수 있다.**

### ptmalloc2 특징
위에서 언급한 것 처럼 `ptmalloc2`의 특징은 다음과 같다.
* 메모리 낭비 방지
* 빠른 메모리 재사용
* 메모리 단편화 방지

#### 매모리 낭비 방지
`ptmalloc`은 메모리 할당 요청이 발생하면, 이전에 해제된 메모리 공간 중에서 재사용할 수 있는 공간이 있는지 탐색한다. 해제된 메모리 공간 중에서 요청된 크기와 같은 크기의 메모리 공간이 있다면 재사용한다. 또한, 작은 크기의 할당 요청이 있으면 해제된 메모리 공간 중 요청한 크기보다 큰 영역에서 메모리를 나누어 주기도 한다.

#### 메모리 재사용
메모리를 재사용하려면 해제된 메모리 주소를 알고 있어야 한다. `ptmalloc`은 `tcache` 또는 `bin`이라는 연결 리스트에 해제된 메모리 주소를 저장한다.

#### 메모리 단편화 방지
메모리 단편화(Memory Fragmentation)는 내부 단편화와 외부 단편화가 있다.

* **내부 단편화**: 할당한 메모리 공간의 크기에 비해 실제 데이터가 점유하는 공간이 작을 때 발생한다.
* **외부 단편화**: 할당한 메모리 공간들 사이에 공간이 많아서 발생하는 비효율을 의미한다.

메모리 단편화가 심해질수록, 각 데이터 사이에 공간이 많아져서 메모리 사용의 효율이 감소한다. 이를 해결하고자 `ptmalloc`은 **정렬(Alignment)**, **병합(Coalescence)**, **분할(Split)**을 사용한다. 

* 정렬 : 64비트 환경에서 메모리 공간을 16바이트 단위로 할당
* 병합 : 특정 조건을 만족하면 해제된 공간들을 병합
* 분할 : 병합된 메모리 공간이 이후의 할당 요청에 의해 분할

## ptmalloc 객체
`ptmalloc2`는 **<U>청크(chunk), bin, tcache, arena</U>**를 주요 객체로 사용한다.

### 청크
**청크(Chunk)**는 덩어라리는 뜻으로, 힙을 관리하는 하나의 단위이다. 청크는 헤더와 데이터로 구성된다. **헤더**는 청크의 상태를 나타내고, **데이터**는 사용자가 입력한 데이터가 저장된다.

![Memory Allocator](/assets/img/ptmalloc2/Memory Allocator.png)

헤더는 청크의 상태를 나타내므로 사용 중인 청크와 해제된 청크의 헤더 구조는 다르다. 사용 중인 청크는 `fd`와 `bk`를 사용하지 않고, 그 영역에 사용자가 입력한 데이터를 저장한다.

청크 헤더의 각 요소는 다음과 같다.

| 이름         | 크기  | 의미                                |
|:-------------|-------|------------------------------------|
|**prev_size** | 8byte | 이전 chunk의 szie                   |
|**size**      | 8byte | chunk의 size                        |
|**fd**        | 8byte | free된 chunk에만 세팅됨, 연결 리스트에서 다음 청크를 가리킴 |
|**bk**        | 8byte | free된 chunk에만 세팅됨, 연결 리스트에서 이전 청크를 가리킴 |

* **A, M, P**: 현재 chunk의 `flag`
    * **A(Allocated Arena, 0x4)**: main	arena가 아닌 arena에서 할당 됐을 경우 세팅
    * **M(Mmap'd, 0x2)**: chunk가 mmap() 함수로 할당 받았다면 1로 세팅
    * **P(Prev In Use, 0x1)**: 인접한 이전 chunk가 사용되거나 free된 chunk가 fastbin에 있으면 세팅

* **Top chunk**
    * 힙의 맨 마지막에 존재하는 chunk
    * 새롭게 malloc을 통해 할당하면 top chunk에서 분리되어 chunk를 할당함
    * Top chunk가 분리되면 두 개의 chunk로 나뉨
        * User chunk: 사용자가 요청한 크기
        * Reminder chunk: 요청한 크기의 나머지 부분이 새로운 top chunk

### bin
`bin`은 사용이 끝난 청크들을 효율적으로 관리하기 위한 자료구조로, 사용이 끝난 청크들이 저장되는 객체이다. 

bin은 사용이 끝난 청크들을 **크기** 단위로 관리한다.`ptmalloc`에는 128개의 bin이 정의되어 있다. 62개는 `smallbin`, 63개는 `largebin`, 1개는 `unsortedbin`으로 사용되고, 나머지 2개는 사용되지 않는다.

`bin`을 이용해 청크들을 재활용하여 메모리를 효율적으로 관리한다.

#### fastbin
fastbin의 특징은 다음과 같다.
* 메모리 할당과 해제가 가장 빠른 bin
* 32 ~ 176byte 까지의 크기가 관리되며, 16바이트 단위로 총 10개의 fastbin이 있다. 리눅스는 작은 크기부터 7개의 fastbin만 사용한다. 
    * 32 ~ 128byte(Linux)
    * 단일 연결리스트(`fd`만 사용)
    * LIFO 방식으로 관리

#### smallbin
smallbin의 특징은 다음과 같다.
* 32 ~ 1023byte 까지의 size를 관리(일부 `fastbin`과 겹침)
* 62개의 smallbin 존재
* 하나의 smallbin에는 같은 크기의 청크들만 보관되며, idx가 증가하면 저장되는 청크들의 크기는 16바이트씩 커짐
* 이중 연결 리스트(`fd`, `bk`를 모두 사용)
* FIFO 방식으로 관리

> `unlink`: 청크를 추가하거나 꺼낼 때 연결고리(이중 연결 리스트)를 끊는 과정
{: .prompt-info }

#### largebin
* 1024byte 이상 크기를 관리(일부 `fastbin`과 겹침)
* 63개의 largebin 존재
* 일정 범위안의 크기를 갖는 청크들 모두 보관
* 이중 연결 리스트(`fd`, `bk`를 모두 사용)
* FIFO 방식으로 관리

#### unsortedbin
* 분류되지 않은 청크들을 보관
* 1개만 존재
* `fastbin`에 들어가지 않는 모든 청크들은 해제되면 크기를 구분하지 않고 `unsortedbin`에 보관
* 이중 연결 리스트이며 내부적으로 정렬 X
* FIFO 방식으로 관리

### arena
`arena`는 `bin`의 정보를 모두 담고 있는 객체이다. 

* 레이스 컨디션을 피하기 위해 최대 64개 arena 생성 가능

### tcache
`tcache`(thread local cache)는 각 쓰레드에 독립적으로 할당되는 캐시이다. 

* glibc 2.26 도입
* 32 ~ 1040 byte 까지의 size를 관리(가장 먼저 조회)
* 각 쓰레드는 64개의 tcache를 갖는다.
* 하나의 tcache는 같은 크기의 청크들만 보관(한 tcache에 최대 7개 청크 보관)
* 보관될 tcache가 가득찼을 경우는 적절한 bin으로 분류
* 단일 연결 리스트
* LIFO 방식으로 관리
